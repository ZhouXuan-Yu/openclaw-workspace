# 内审 Skill — 自我审查协议

## 触发条件

用户说"内审"、"审查"、"反思"、"复盘"、"质量检查"时触发。

## 内审维度（5 大模块）

### 模块 1：内容质量审查

**检查项**：
- [ ] 内容是否基于真实经历/数据（不编造、不夸大）
- [ ] 技术细节是否准确（代码、架构、工具用法）
- [ ] 是否有错误归因（如把别人的工作说成自己的）
- [ ] 文字是否简洁有力（去除废话和套话）
- [ ] 格式是否符合目标平台（小红书无Markdown、B站标题党等）
- [ ] 是否有错别字、语法错误
- [ ] 图片/视频质量是否达标

**评分标准**：
| 等级 | 条件 |
|------|------|
| ✅ 通过 | 无硬伤，可发布 |
| ⚠️ 需修改 | 有小问题，修改后可发布 |
| ❌ 打回 | 有硬伤（编造、错误归因、严重技术错误），必须重写 |

### 模块 2：人设一致性审查

**检查项**：
- [ ] 内容是否符合"Agent工程开发者"定位
- [ ] 是否体现"从零到一"的真实学习过程
- [ ] 是否保持"技术深度+实战经验"的风格
- [ ] 是否避免"卖课感"、"鸡汤感"、"标题党过度"
- [ ] 是否保持原创性（不抄袭、不洗稿）
- [ ] 是否体现个人思考（不只是搬运信息）
- [ ] 人设是否有漂移（从技术分享变成生活博主等）

**人设锚点**：
```
我是谁：一个在Agent工程领域从零到一的开发者
我在做什么：记录真实学习过程，分享踩坑经验
我的价值：让后来者少走弯路，和同行交流碰撞
我的底线：不编造、不夸大、不卖课、不割韭菜
```

### 模块 3：平台策略审查

**检查项**：
- [ ] 各平台内容是否差异化（不是简单复制粘贴）
- [ ] 发布频率是否稳定（不能断更太久）
- [ ] 各平台数据是否在跟踪（播放、点赞、收藏、评论）
- [ ] 是否有爆款内容可复用
- [ ] 是否有低效内容需调整
- [ ] 互动是否及时（评论回复、私信处理）

**平台矩阵**：
| 平台 | 定位 | 内容形式 | 频率 |
|------|------|---------|------|
| 小红书 | 技术干货+学习笔记 | 图文 | 2-3篇/周 |
| 抖音 | 短视频教程+踩坑记录 | 视频15-60s | 2-3条/周 |
| B站 | 深度教程+项目实战 | 视频5-15min | 1-2条/周 |
| 快手 | 技术分享+日常 | 图文/视频 | 2-3篇/周 |

### 模块 4：学习进度审查

**检查项**：
- [ ] 本周学了什么新东西
- [ ] 是否有可输出的学习成果
- [ ] 是否有踩坑经验可分享
- [ ] 是否有技术突破可记录
- [ ] 学习方向是否偏离主线
- [ ] 是否有知识盲区需补强

**学习主线**：
```
Agent工程化 → 工具链搭建 → 自动化流水线 → 内容生产 → 个人IP
```

### 模块 5：风险与合规审查

**检查项**：
- [ ] 是否泄露敏感信息（API Key、密码、内部文档）
- [ ] 是否涉及版权问题（代码、图片、文章）
- [ ] 是否有法律风险（虚假宣传、侵权）
- [ ] 是否有平台违规风险（引流、营销过度）
- [ ] 是否有隐私风险（他人信息、聊天记录）
- [ ] 是否有安全风险（暴露端口、凭证）

## 内审流程

### Step 1：数据收集
```powershell
# 读取今日生产内容
Get-ChildItem "C:\Users\ZhouXuan\Desktop\Agent每日生产（重要）" -Recurse

# 读取今日记忆日志
Read-Host "memory/2026-MM-DD.md"

# 读取平台数据（如有）
```

### Step 2：逐项审查
按 5 个模块逐一检查，给出评分和问题清单。

### Step 3：问题分类
| 类型 | 处理方式 |
|------|---------|
| 致命问题 | 立即停止发布，修复后重审 |
| 严重问题 | 修改后重审 |
| 一般问题 | 记录，下次改进 |
| 建议优化 | 记录，择机改进 |

### Step 4：输出内审报告

**报告格式**：
```markdown
# 内审报告 — YYYY-MM-DD

## 总评
- 评分：X/100
- 状态：✅ 通过 / ⚠️ 需修改 / ❌ 打回

## 问题清单
| # | 模块 | 问题 | 严重性 | 处理 |
|---|------|------|--------|------|
| 1 | 内容质量 | xxx | 严重 | 修改 |
| 2 | 人设一致性 | xxx | 一般 | 记录 |

## 改进建议
1. xxx
2. xxx

## 下次审查重点
- xxx
```

### Step 5：写入记忆
将内审结果写入 `memory/daily/` 和 `memory/topics/`。

## 快速内审（轻量版）

当时间紧迫时，只做核心检查：

**3 个必查项**：
1. 内容是否真实？（不编造）
2. 人设是否一致？（不漂移）
3. 有无风险？（不违规）

**1 个必问问题**：
> 如果有人按这个内容去操作，会不会出问题？

## 深度内审（完整版）

当需要全面审查时，执行完整 5 模块检查，并输出详细报告。

适用场景：
- 月度/季度复盘
- 重大内容发布前
- 人设调整前
- 平台策略变更前

## 内审频率

| 类型 | 频率 | 范围 |
|------|------|------|
| 快速内审 | 每次发布前 | 3 个必查项 |
| 标准内审 | 每周一次 | 5 模块完整检查 |
| 深度内审 | 每月一次 | 全面审查+策略调整 |

## 注意事项

1. **内审不是自我否定** — 是质量控制，不是自我批评
2. **问题要具体** — 不说"内容不好"，说"第3段技术细节有误"
3. **改进要可执行** — 不说"下次注意"，说"下次发布前先跑测试用例"
4. **保持客观** — 不因为是自己的内容就放松标准
5. **记录教训** — 每次内审的教训都要沉淀到记忆系统
