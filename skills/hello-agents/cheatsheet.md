# Cheatsheet

## 范式选择矩阵

| 场景 | 范式 | 理由 |
|------|------|------|
| 需要外部知识/实时信息 | ReAct | 边想边做，动态调用工具 |
| 强结构可分解任务 | Plan-and-Solve | 先规划后执行，目标一致 |
| 需要迭代改进输出 | Reflection | 生成→批判→修正 |
| 长流程+多工具 | ReAct | 走一步看一步，灵活纠错 |
| 数学/逻辑严谨任务 | Plan-and-Solve | 分步推理，减少出错 |

## 工具描述模板
```
工具名称: 简洁唯一标识符
工具描述: 清晰的何时用自然语言描述(←最关键)
执行逻辑: 函数/方法实现
```

## 评估基准速查

| 基准 | 样本量 | 评估维度 | 方法 |
|------|--------|----------|------|
| BFCL | 1120+ | 工具调用精度 | AST 匹配 |
| GAIA | 466 | 综合问题解决 | 准精确匹配 |
| ToolBench | 16000+ | API 调用 | 多维度 |
| AgentBench | 8 领域 | 通用能力 | 任务完成度 |

## 上下文工程阈值

| 参数 | 建议值 | 说明 |
|------|--------|------|
| ReAct max_steps | 5-15 | 防止无限循环 |
| Temperature | 0 | Agent 需要确定性 |
| 上下文窗口使用率 | <70% | 为工具返回留空间 |
| MVTS 工具数 | 3-8 | 超过则 LLM 选择困难 |
| SFT 数据量 | 1000-10000 | 对 (prompt, completion) |
| 评估预算/次 | $10-50 | API 调用成本 |

## Protocol 选择
- 需要标准化调用工具 → **MCP**
- 需要多 Agent 对话 → **A2A**
- 需要大规模服务发现 → **ANP**
