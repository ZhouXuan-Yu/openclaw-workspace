# 第9章 上下文工程

## Core Idea
上下文工程是提示工程的演进。核心不是"怎么写提示"，而是"在有限的注意力预算下，策划最优信息集合(tokens)进入上下文窗口"。

## Frameworks Introduced
- **GSSC 流水线**: Gather-Select-Structure-Compress — 上下文构建四步法
- **ContextBuilder**: HelloAgents 上下文构建器
- **NoteTool**: 结构化笔记工具，持久化记忆管理
- **TerminalTool**: 文件系统操作和即时上下文检索

## Key Concepts
- **上下文工程 vs 提示工程** — 从"怎么写提示"到"策划整个信息集合"
- **上下文腐蚀 (Context Rot)** — 上下文增长后模型准确回忆能力下降
- **注意力预算** — 每新增 token 消耗注意力预算
- **最小必要信息集** — 勾勒期望行为的最少 tokens
- **最小可行工具集 (MVTS)** — 精心甄别而非塞入全部工具
- **及时上下文 (JIT Context)** — 维护引用，运行时动态加载
- **渐进式披露** — 每步交互产生新上下文指导下一步

## Mental Models
1. **信息是有限资源**: 上下文窗口有边际收益递减
2. **JIT 上下文**: 维护引用而非数据本身
3. **针堆找针**: 上下文越长越难找到关键信息

## Anti-patterns
- 臃肿工具集(边界模糊的工具越多，选错概率越高)
- 过度硬编码提示
- 一次性塞入所有数据

## Code Examples
```
from hello_agents.context import ContextBuilder
builder = ContextBuilder()
context = builder.gather(system_prompt, tools, memory, observations)\
                 .select(key_fields=["task", "current_step"])\
                 .structure(format="markdown")\
                 .compress(max_tokens=4096)
```

## Worked Example
**JIT 上下文处理代码仓库**: 先获取目录树(引用) → 发现 src/core/ 是关键 → 动态加载 agent.py → 按需加载 llm.py。保证上下文被最高信号密度的信息填满。

## Key Takeaways
1. 上下文是有限预算，每 token 都要精打细算
2. GSSC 提供可复用的上下文管理方法论
3. JIT 上下文更贴近人类认知
4. MVTS 提升 Agent 长期稳定性
5. 好示例胜过罗列所有边界条件
