# 循环池 — Progress Heartbeat

> 解决问题：长时间执行任务时，用户不知道 Agent 是否断线
> 目标：定期发送进度更新，让用户知道"我还在工作"

## 问题分析

当前痛点：
- Agent 执行复杂任务时（研究、多步操作），可能 5-10 分钟无输出
- 用户无法区分：正在工作 vs 断线卡死 vs 忘记回复
- 用户焦虑感：不知道该等还是该重试

## 设计方案

### 方案 A：子代理 + 主代理轮询

```
用户请求（复杂任务）
    ↓
主 Agent 识别为长任务
    ↓
spawn 子代理执行实际工作
    ↓
主 Agent 每 60 秒轮询子代理状态
    ↓
主 Agent 发送进度更新给用户
    ↓
子代理完成 → 主 Agent 汇总结果
```

优点：主 Agent 始终在线，可以响应用户中断
缺点：多一层开销，token 成本增加

### 方案 B：分步执行 + 每步汇报

```
用户请求（复杂任务）
    ↓
主 Agent 拆解为 N 个步骤
    ↓
执行 Step 1 → 汇报进度
执行 Step 2 → 汇报进度
...
执行 Step N → 汇报结果
```

优点：简单直接，无额外开销
缺点：不适合单步长时间任务（如深度研究）

### 方案 C：混合方案（推荐）

```
用户请求
    ↓
判断预估耗时
    ├── <30 秒 → 直接执行，不需要循环池
    ├── 30秒-2分钟 → 分步汇报（方案 B）
    └── >2分钟 → spawn 子代理 + 轮询汇报（方案 A）
```

## 循环池消息模板

### 进度更新格式

```
⏳ 进度 [步骤N/总计]
├── 当前: [正在做什么]
├── 已完成: [完成了什么]
└── 预计: [还需要多久]
```

### 示例

```
⏳ 进度 [2/5]
├── 当前: 搜索 OpenClaw 安全文档
├── 已完成: 分析当前安全配置
└── 预计: 还需 ~2 分钟
```

### 完成格式

```
✅ 任务完成
├── 耗时: 3 分 42 秒
├── 步骤: 5/5
└── 结果: [简要总结]
```

### 错误格式

```
⚠️ 任务受阻
├── 步骤: 3/5
├── 问题: [具体问题]
└── 方案: [正在尝试的解决方案]
```

## 触发规则

| 条件 | 操作 |
|------|------|
| 任务预估 <30秒 | 不启用循环池 |
| 任务预估 30秒-2分钟 | 每 30 秒汇报一次 |
| 任务预估 >2分钟 | spawn 子代理 + 每 60 秒轮询汇报 |
| 步骤切换时 | 立即汇报（不等定时器） |
| 遇到错误时 | 立即汇报 |
| 用户主动问 | 立即回复当前状态 |

## 实现方式

### 在 AGENTS.md 中添加规则

在执行-验证循环中加入：
```
长任务识别 → 启用循环池 → 定期汇报 → 完成/阻塞时汇报
```

### 子代理任务模板

```
你是执行子代理。执行以下任务：
[任务描述]

每完成一个步骤，更新状态文件：
状态文件: .tmp/progress-{task_id}.json

格式:
{
  "step": 2,
  "total": 5,
  "current": "搜索安全文档",
  "completed": ["分析当前配置"],
  "eta_seconds": 120,
  "status": "running"
}
```

### 主代理轮询模板

```
检查进度文件 .tmp/progress-{task_id}.json
如果更新了 → 发送进度给用户
如果 60 秒没更新 → 发送"仍在处理中..."
如果子代理完成 → 汇总结果发送
```

## 消息频率控制

| 阶段 | 频率 | 原因 |
|------|------|------|
| 开始 | 立即 | 确认收到任务 |
| 执行中 | 每 30-60 秒 | 保持用户感知 |
| 步骤切换 | 立即 | 重要节点 |
| 遇阻 | 立即 | 用户可能需要干预 |
| 完成 | 立即 | 交付结果 |

## 安全边界

- 循环池消息不包含敏感信息
- 不在群聊中发送进度更新（只在私聊）
- 用户说"安静"或"别发了"时停止循环池
- 循环池消息不计入记忆系统
